SeaClouds Platform

Summary on the different capabilities implemented by SeaClouds Platform.

This document guides the user through the SeaClouds Dashboard Demo, describing the architecture of SeaClouds Platform and the workflow managed by each of the different components.

Home

This is a proof of concept of the SeaClouds Dashboard as result of the early prototype of the First Year of SeaClouds Platform.

This current version offers a limited functionality of what expected from the finished product in the SeaClouds project, containing:

  • a prototype on the Module Profile Designer, where the user can design the topology of the application,
  • a first version of the Matchmaking process (in the Discoverer & Planner components),
  • Deployment at IaaS level (in the Deployer component),
  • Monitoring mechanism to check sensors/information of the deployed application (in the Monitor component),
  • and a first version of the SLA management checking QoS violations (in the SLA Service omponent).

Module Profile Designer

In this section, the user can easily design the topology of the application by dragging modules and configure them. This topology should include dependencies and connections between modules (e.g. a Java web application needs access to an existing MySQL database).

Once the application topology has been defined, the designer will generate an Abstract Application Model using OASIS TOSCA standard. This model will be passed to the next component: Discoverer & Planner.

Discoverer & Planner

In this phase, the user can modify (if needed) the generated Abstract Application Model using the YAML editor. This plan describes the multiple requirements and constraints on the modules that compose the application as well as some of the initial configuration.

This model is then processed by the Planner, which will fill in the application configuration with the best configuration suitable for the specified constraints. This process of finding the best configuration is performed by the Optimizer.

Once this process is completed, a Deployable Application Model (DAM), following the OASIS CAMP standard), is generated and ready to be loaded by the SeaClouds Deployer.

Deployer

Next phase involves deploying the application. This task is initialized by the previosly generated Deployable Application Module, and it contains the final specification for the application configuration and topology.

SeaClouds Deployer is built over Apache Brooklyn, a library and control plane for deploying and managing distributed applications. More details can be found here.

SeaClouds Deployer manages the deployed applications, allowing the user to execute management actions such as start/stop/restart on the application modules, explore the topology of the application or keep track of the status on the deployment phase.

Monitor

In this section, the user perform a more detailed control on the behaviour of the application. The user can select one of the deployed applications, choosing from a list of metrics and drawing graphs that will be constantly updated with live data from the managed system.

SLA Service

SeaClouds Platform also allows users to create and control SLA agreements with cloud providers. Using the SLA Service users can create a new SLA agreement that will guarantee some condition over the system and alert users when the agreements are violated.
